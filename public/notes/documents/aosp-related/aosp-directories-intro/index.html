AOSP directories intro

The ‚Äã‚Äã‚ÄãAndroid Open Source Project (AOSP)‚Äã‚Äã‚Äã is a large-scale project, and its source code is organized into various directories. Each directory plays a specific role in the Android platform’s development. Below is a high-level overview of the main directories within AOSP:

1. art/

  • Purpose: Contains the Android Runtime (ART), which is responsible for executing Android applications. ART is the runtime that replaced the Dalvik VM, supporting Ahead-of-Time (AOT) and Just-in-Time (JIT) compilation.
  • Key subdirectories:
    • compiler/: Handles the AOT/JIT compilation of bytecode.
    • runtime/: Core runtime engine that executes the compiled code.

2. bionic/

  • Purpose: Implements the Android version of the standard C library (libc), optimized for low memory and performance in embedded systems. It also includes the dynamic linker (libdl) and the math library (libm).
  • Key subdirectories:
    • libc/: The C standard library.
    • libm/: The math library.
    • linker/: The dynamic linker responsible for loading shared libraries.

3. bootable/

  • Purpose: Contains components for the boot process of Android devices, such as the recovery system.
  • Key subdirectories:
    • recovery/: Source for the Android recovery system, used for flashing, updating, or resetting the device.

4. build/

  • Purpose: This directory manages the build system for Android, containing scripts and configuration files that define how the entire Android platform is compiled.
  • Key subdirectories:
    • core/: Core makefiles and scripts for the build process.
    • soong/: Newer build system based on Soong, which is gradually replacing the older make system.

5. cts/

  • Purpose: Houses the Compatibility Test Suite (CTS), used to ensure that devices comply with Android‚Äôs compatibility requirements.
  • Key subdirectories:
    • tests/: The actual test cases for various components of Android.

6. dalvik/

  • Purpose: Contains the legacy Dalvik VM, the runtime environment that was used before ART. Although no longer actively used, this directory holds the legacy code for reference.
  • Key subdirectories:
    • vm/: The Dalvik virtual machine.

7. development/

  • Purpose: Contains tools and utilities for Android developers, such as sample code, development tools, and debugging utilities.
  • Key subdirectories:
    • tools/: Various development tools and scripts.
    • samples/: Sample applications and framework examples.

8. device/

  • Purpose: Contains device-specific configurations, typically used to define device hardware characteristics, build configurations, and board support packages (BSPs).
  • Key subdirectories:
    • ‚Äãgeneric/‚Äã: Generic configurations for reference devices.
    • ‚Äãgoogle/‚Äã: Device-specific configurations for Google-branded devices like Pixel.
    • ‚Äãqcom/‚Äã: Qcom defined SSI
    • ‚Äãxiaomi/missi/‚Äã: Extend SSI of xiaomi from Qcom SSI
    • ‚Äãxiaomi/xxx/product/‚Äã: xxx’s product

9. external/

  • Purpose: Includes external open-source libraries and projects that Android depends on but are not natively developed by the Android team.
  • Key subdirectories:
    • Multiple external libraries like openssl/, sqlite/, webkit/, etc.

10. frameworks/

  • Purpose: Contains the core framework that defines Android‚Äôs application model and essential system services. It‚Äôs one of the most critical directories in AOSP.
  • Key subdirectories:
    • base/: Core Android framework, containing essential services like Activity Manager, Package Manager, etc.
    • av/: Audio/Video framework, handles media playback, recording, and camera access.
    • native/: Native system framework that handles low-level services such as SurfaceFlinger for rendering.
    • opt/: Optional framework components such as Bluetooth and telephony.

11. hardware/

  • Purpose: Contains hardware abstraction layers (HALs) for different devices, allowing Android to interact with hardware like cameras, sensors, and radios.
  • Key subdirectories:
    • libhardware/: Generic hardware abstraction library.
    • interfaces/: HAL interfaces for different hardware components.

12. kernel/ (Optional, not always part of AOSP)

  • Purpose: Contains device-specific Linux kernel sources. Not every AOSP repository includes the kernel, as it is typically distributed separately.
  • Note: This directory may vary depending on the device and is often provided by the device manufacturer.

13. libcore/

  • Purpose: Provides core libraries for Android, including the Java class libraries (like java.util, java.lang) and APIs that Android applications depend on.
  • Key subdirectories:
    • ojluni/: OpenJDK-based libraries used by Android for its Java runtime.

14. packages/

  • Purpose: Contains default system applications and services that come with Android, like the launcher, calendar, contacts, and messaging apps.
  • Key subdirectories:
    • apps/: Core Android applications (e.g., Dialer, Settings).
    • providers/: Content providers (e.g., ContactsProvider, MediaProvider).
    • services/: System services that interact with apps and hardware.

15. platform_testing/

  • Purpose: Contains test suites and testing frameworks to ensure that the platform functions as expected. This is vital for regression testing.
  • Key subdirectories:
    • tests/: Test cases for platform functionality.
    • cts/: Compatibility Test Suite cases for platform compliance.

16. prebuilts/

  • Purpose: Houses precompiled binaries and libraries used during the build process, such as toolchains, compilers, and prebuilt SDK components.
  • Key subdirectories:
    • sdk/: Prebuilt SDK libraries for Android development.
    • gcc/: Prebuilt toolchains for compiling Android.

17. sdk/

  • Purpose: Contains components for the Android Software Development Kit (SDK), such as the Android emulator and tools for application development.
  • Key subdirectories:
    • tools/: Android SDK tools like adb and emulator.

18. system/

  • Purpose: Contains the core system components needed for Android‚Äôs runtime and basic system services.
  • Key subdirectories:
    • core/: Core system services like init, logcat, adb.
    • sepolicy/: SELinux policy configuration.
    • vold/: Volume daemon for managing storage.

19. test/

  • Purpose: Contains tests for various parts of the system, particularly focusing on lower-level functionalities.
  • Key subdirectories:
    • jni/: Tests for native code (JNI).
    • suites/: Organized test suites.

20. tools/

  • Purpose: Provides a range of utilities and tools for building and debugging Android, such as the Android Debug Bridge (adb), the TraceView profiler, and others.
  • Key subdirectories:
    • adb/: Source code for the adb tool.
    • traceview/: Tools for profiling and performance analysis.

21. vendor/

  • Purpose: Contains vendor-specific code, such as proprietary drivers, HAL implementations, and device-specific configurations provided by device manufacturers.
  • Key subdirectories:
    • Vendor-specific configurations and binaries.
