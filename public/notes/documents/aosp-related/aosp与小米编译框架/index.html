AOSP与小米编译框架

小米自有编译系统为盘古架构, 本文档主要描述当前盘古架构的编译系统架构方案以及修改SOC的AOSP代码为盘古架构的步骤和约束等.

OS模块化

减少 system/vendor 镜像的变体，让更多的项目可以共用相同的 system/vendor 镜像，把区域、产品形态等差异进一步解耦到 system_ext/odm/product 分区，实现 system/vendor 的独立编译，大幅提升 OS 编译速度。

System独立编译

System 与 System_ext 主要托管 Java 与 C/C++ 程序，System 与 System_ext 的解耦，主要是其中的 Java、C/C++ 程序解耦，所以 CSI 本质是上述两种语言的稳定接口层：

  • Java Layer
  • Android Internal SDK，Android Fwk 提供给 MIUI Fwk 的 Java 稳定接口层
  • MIUI Ext SDK，MIUI Fwk 提供给 Android Fwk 的 Java 稳定接口层
  • C/C++ Layer
  • Android Internal NDK，Android Fwk 提供给 MIUI Fwk 的 C/C++ 稳定接口层
  • MIUI Ext NDK，MIUI Fwk 提供给 Android Fwk 的 C/C++ 稳定接口层

当 System 与 System_ext 中的程序可以通过上述稳定接口层独立编译后，在运行时还需要通过动态加载框架把不同镜像的程序动态链接，以及管控其实例的生命周期：

  • Java Dynamic Frameowrk，动态加载 .jar 库并创建、关联、管理实例
  • C/C++ Dynamic Framework，动态链接 .so 库并创建、关联、管理实例

Vendor独立编译

实现 Vendor 镜像编译时无需依赖 System Source Tree，依赖 SoC 支持。

仓库与分支模型

基于不同的分支模型，可以把仓库分为以下几类，主线制分支模型主要应用于SSI、Vendor类型的仓库：
仓库类型 说明 包含模块示例 分支模型
SSI 系统框架、服务等不区分机型的通用功能 Framework base，service等系统层模块 按芯片厂商、Andorid大版本共用的主线分支，如master-u-qcom，master-u-mtk；按芯片厂商共用的主线分支，如master-qcom，master-mtk；按Android大版本共用的主线分支，如master-u；所有版本通用的主线分支，master。
Vendor 底层BSP不区分机型的通用功能 BT、WIFI、GKI等 允许使用SSI的分支模型，除此以外：按芯片平台划分的仓库，如master-u-qcom-sm8550；按内核版本划分的仓库，如master-u-mtk-4.14;
Device 区分机型的代码仓库，包括：上层product仓库，底层BSP区分机型的仓库等 product分区相关代码，camera相关代码等 可以按机型、SKU维度拉分支。
Non-hlos modem、sensor等在专有芯片上运行的代码仓库 modem、sendor、AOP、NPU等 可以按照机型+region的维度拉分支。

组件和仓库分类的关系

仓库分类是基于分支模型划分的；组件指可独立编译的镜像。

仓库分类和组件是多对多的关系，一个分类里可以包含多个组件，比如Device分类包含product、odm等；多个分类的仓库也可能会编译到一个组件里，比如属于Device分类的camera模块、属于Vendor分类的底层通用功能，都会编译到vendor组件。

主线分支命名规范
仓库类型 复用范围 分支名称规则
1 通用代码仓库 对外部实现没有依赖，完全通用，只创建一个主线分支，master。
2 同芯片厂商共线的仓库 每个芯片厂商一个主线分支。 命名规则：master-[芯片厂商]，master-qcom，master-mtk
3 同一个Android大版本上共线的仓库 每个Android大版本一个主线分支。主线分支命名规则：master-[Android大版本代号]，例如：master-t，master-u。
4 同Android版本、同芯片厂商共线的仓库 根据Android版本和芯片厂商建立不同的主线分支。主线分支命名规则：master-[Android大版本]-[芯片厂商]例如：master-t-qcom，master-t-mtk
5 同芯片平台共线的仓库 一个芯片平台建立一个主线分支。主线分支命名规则：master-[Android大版本代号]-[芯片厂商]-[芯片平台]，例如：master-t-qcom-sm8550，master-t-mtk-mt6985。
6 区分机型的仓库 基于不同机型拆分仓库代码。分支命名规则：master-[Android大版本]–[芯片厂商]-[芯片平台]-[device名称]例如：master-t-qcom-sm8550-nuwa
临时分支
分支类型 作用 进版要求 测试活动 生命周期
保密集成分支 用于MIUI大版本保密功能的持续集成。保密功能开发完成后merge到这个分支，用于保密版本日构建打包，按规划拉出保密版本release分支。从master主线分支的相关仓库拉出，每天自动同步master主线分支上的修改。 同master主线分支 根据保密需求，在MIUI大版本TR4前从master拉出，发布会后merge回master主线分支。
Feature分支 开发分支，用于机型适配（TR4前），基线升级，新功能开发等场景。保密feature分支用于MIUI大版本严格保密功能的开发，设置权限控制。从master主线分支的相关仓库拉出。 业务组自行管控 merge回主线时要求通过相关功能的验收测试、CI/CT测试； 跟随Feature的开发周期，分支和Feature JIRA ID绑定。Feature开发完成merge回主线后，关闭对应分支（不能继续提交）。
Hotfix分支 基于某个特定版本，添加紧急修复代码发版本。基于固定的代码节点拉出（比如基于某个日构建），拉出后不再更新base代码。 SPM审批进版。 紧急修复验收测试 按需拉，完成发版后关闭分支。
Feature分支

![](assets/Pasted image 20241101163621-20250311094648-8wv3ukc.png )

盘古3.0规范在Android 升U和MIUI15开发上落地，U的项目都要采用feature分支来进行机型适配、基线升级和新功能开发。

  • master-u主线分支禁止直接push Feature相关代码（通过change相关的JIRA判断），必须使用Feature分支进行开发。

  • Feature分支和一个Feature JIRA绑定，仅用于该feaute的开发、自测。此分支对研发开放所有权限，没有CI/CT检查，方便业务组快速修改代码，业务组研发自己保证代码质量。

  • Feature分支完成开发后，必须通过验收测试和CI/CT测试后才能合入主线。

    • 机型适配和基线升级根据预先设计的case集进行验收，新功能根据TR4设计的测试方案来验收。

Manifest仓库配置

为了让各个组件的逻辑关系更清晰并且解决很多开发人员不知道代码应该合入SSI还是Vendor的问题，所以对Manifest按照盘古架构进行了相应的拆分，将Manifest文件拆分为SSI，Vendor，Device 和 BP (NONHLOS)几个部分，下载代码前需要，将这几部分组合生成完整的manifest才能完成代码的下载。

mibuild.py

mibuild.py 为编译的入口，用户通过调用mibuild，完成构建。

mibuild中会调用build-system-pipeline.sh，build-product-pipeline.sh，build-vendor-pipeline.sh，build-odm-pipeline.sh 等，生成各个组件的targetfiles.zip。

再调用 merge-all-target-files-pipeline.sh 完成各个组建targetfiles的merge，并且生成最终需要的targetfiles.zip。

最后调用sign-target-files-pipeline.sh完成最终targetfiles.zip的签名。
